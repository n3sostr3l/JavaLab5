# Вопросы к защите лабораторной работы №6

## 1. Сетевое взаимодействие - клиент-серверная архитектура, основные протоколы, их сходства и отличия
**Клиент-серверная архитектура** — вычислительная архитектура, в которой система разделена на поставщиков ресурсов/услуг (серверы) и заказчиков (клиентов). Сервер выполняет основную логику и хранит данные, клиент отправляет запросы на выполнение команд.
**Сходства TCP и UDP**: Оба протокола работают на транспортном уровне модели OSI, передают данные по сети от одного порта к другому, используют IP-адреса.
**Отличия**: 
- **TCP** — протокол с установлением соединения, гарантирует доставку и порядок пакетов (использует подтверждения), более надежный, но медлительный.
- **UDP** — протокол без установления соединения, отправляет датаграммы без гарантии доставки и порядка, работает быстрее, подходит для транслирования видео/аудио.

**Пример из кода:** В нашем приложении используется клиент-серверная архитектура на базе TCP. Клиент (`NetworkManager.java`) отправляет запросы, а сервер (`ServerManager.java`) слушает порт, обрабатывает их и шлёт ответы.

## 2. Протокол TCP. Классы Socket и ServerSocket
В Java для блокирующего взаимодействия по TCP в пакете `java.net` есть два основных класса:
- `Socket` — это конечная точка (клиентская часть), с помощью которой устанавливается соединение с сервером.
- `ServerSocket` — класс сервера, который «слушает» определённый порт и ждёт подключений клиентов через метод `accept()`.

**Пример из кода (клиентская часть):**
```java
// NetworkManager.java
socket = new Socket(host, port);
out = new DataOutputStream(socket.getOutputStream());
in = new DataInputStream(socket.getInputStream());
```

## 3. Протокол UDP. Классы DatagramSocket и DatagramPacket
- `DatagramSocket` — класс, представляющий сокет для отправки и приёма датаграммных пакетов поверх UDP. Он не устанавливает выделенного соединения.
- `DatagramPacket` — класс, представляющий сам пакет данных, который включает в себя массив байт, длину, а также (при отправке) IP-адрес и порт получателя.

*Примечание: В нашем проекте шестая лабораторная реализована через потоковый TCP (с помощью NIO на сервере и блокирующих сокетов на клиенте), поэтому UDP классов в чистом виде в нашем коде нет.*

## 4. Отличия блокирующего и неблокирующего ввода-вывода, их преимущества и недостатки. Работа с сетевыми каналами
**Блокирующий ввод-вывод (OIO / java.io):** Поток выполнения программы останавливается (блокируется), пока не будут получены или отправлены данные.
- *Преимущества*: Код пишется просто, последовательно (линейно).
- *Недостатки*: Для каждого клиента требуется свой поток (`Thread`), что потребляет много ресурсов при большом числе подключений. В нашем клиенте отправка ожидаемо блокируется: `in.readFully(responseData)`.

**Неблокирующий ввод-вывод (NIO / java.nio):** Поток не ждет данных, операция дает немедленный результат (если данных нет — возвращается пустой буфер или 0).
- *Преимущества*: Высокая масштабируемость, можно обрабатывать тысячи соединений в одном потоке приложения через Мультиплексор (`Selector`).
- *Недостатки*: Гораздо сложнее в написании, так как данные могут приходить "кусками" (фрагментация). Требуется сохранять состояние чтения.

**Пример из кода:** На сервере мы используем неблокирующий режим с помощью селектора:
```java
// ServerManager.java
serverChannel.configureBlocking(false);
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

// Обработка частично приходящих данных через сохранение ReadState
int read = clientChannel.read(state.dataBuffer);
```

## 5. Классы SocketChannel и DatagramChannel
Эти классы входят в пакет `java.nio.channels` и являются неблокирующими (или блокирующими, по выбору) аналогами `Socket` и `DatagramSocket`.
- `SocketChannel` используется для потоковой передачи данных по TCP (для клиента или принятого соединения на сервере).
- `DatagramChannel` используется для обмена датаграммами по UDP. Ни один из них не привязан к одному конкретному потоку-обработчику; их можно регистрировать в `Selector`.

**Пример из кода:** Сервер использует `SocketChannel` для взаимодействия с подключенными клиентами:
```java
// ServerManager.java - принятие нового клиента
SocketChannel clientChannel = serverChannel.accept();
clientChannel.configureBlocking(false);
```

## 6. Передача данных по сети. Сериализация объектов
Для отправки Java-объектов по сети их необходимо превратить в массив байт (для пересылки по каналам/потокам вывода). Этот процесс называется **сериализацией**. Десериализация — это обратный процесс воссоздания объекта.

**Пример из кода (сериализация и отправка размера пакета):**
```java
// NetworkManager.java - отправка запроса
ByteArrayOutputStream baos = new ByteArrayOutputStream();
ObjectOutputStream oos = new ObjectOutputStream(baos);
oos.writeObject(request); // Сериализация объекта Request
```

**Десериализация на сервере:**
```java
// ServerManager.java - получение запроса
ObjectInputStream ois = new ObjectInputStream(new ByteArrayInputStream(...));
Request request = (Request) ois.readObject(); // Воссоздание объекта Request
```

## 7. Интерфейс Serializable. Объектный граф, сериализация и десериализация полей и методов
Чтобы объект можно было сериализовать, его класс должен имплементировать интерфейс-маркер `java.io.Serializable`. 
- **Объектный граф:** При сериализации объекта автоматически сериализуются все объекты, на которые он ссылается, а также те, на которые ссылаются они, и так далее (т.е. весь граф зависимостей). Если какой-то класс в графе не поддерживает `Serializable`, будет брошено исключение `NotSerializableException`.
- **Поля и методы:** Сериализуются только данные (поля объекта). Статические поля (`static`) не принадлежат объекту и не сериализуются. Поля с модификатором `transient` также игнорируются при сериализации. Методы не сериализуются вообще (на другой стороне объект просто связывается с методами из существующего `.class` файла).

**Пример из кода:** Классы `Request`, `Response`, а также элементы коллекции (`LabWork`, `Coordinates`, `Person` и т.д.) обязательно имплементируют `Serializable`, так как мы передаем их между клиентом и сервером.

## 8. Java Stream API. Создание конвейеров. Промежуточные и терминальные операции
**Stream API** (`java.util.stream`) позволяет в функциональном стиле обрабатывать коллекции данных (фильтровать, сортировать, преобразовывать).
Вычисления состоят из:
1. **Источника** (например, списка элементов).
2. **Промежуточных операций** (Intermediate) — они возвращают новый Stream и являются ленивыми (не выполняются, пока не вызвана терминальная операция). *Примеры*: `filter()`, `map()`, `sorted()`.
3. **Терминальной операции** (Terminal) — завершает конвейер и выдает результат (список, число, отсутствие результата). *Примеры*: `collect()`, `forEach()`, `count()`, `max()`.

**Примеры из кода (в `CollectionManager.java`):**
```java
// Поиск максимального ID (промежуточная: mapToLong, терминальная: max)
labworks.values().stream()
        .mapToLong(LabWork::getId)
        .max()
        .orElse(0L) + 1;

// Фильтрация ключей (промежуточная: filter, терминальная: collect)
java.util.List<Integer> keysToRemove = labworks.keySet().stream()
        .filter(k -> k < key)
        .collect(java.util.stream.Collectors.toList());
```

## 9. Шаблоны проектирования на примерах из Java и нашего кода
- **Command (Команда):** В коде реализуется в виде классов, инкапсулирующих действие. У нас это `ShowCommand`, `UpdateCommand`, которые выполняются через `CommandInvoker`.
- **Iterator (Итератор):** Встроен в коллекции Java, позволяет обходить элементы. В нашем коде используется при обходе селектора NIO: `Iterator<SelectionKey> iter = selectedKeys.iterator();` в `ServerManager.java`.
- **Decorator (Декоратор):** Динамическое добавление нового поведения. Активно используется в Java IO потоках. В нашем коде: `new ObjectInputStream(new ByteArrayInputStream(...))` — `ObjectInputStream` декорирует `ByteArrayInputStream`, добавляя возможность читать целые объекты, а не только байты.
- **Singleton (Одиночка):** Класс, у которого может быть только один экземпляр. В нашем коде `CollectionManager` работает как подобие Singleton: он содержит только статические методы и единое статическое хранилище коллекции `Hashtable`. В Logger'ах: `LogManager.getLogger(...)` также отдает один и тот же логгер для класса.
- **Factory Method (Фабричный метод):** Создание объектов без прямого использования `new`. В коде: статический фабричный метод `ServerSocketChannel.open()` и `Selector.open()`.
- **Strategy (Стратегия):** Различное поведение, подставляемое во время выполнения. В Java часто используется в компараторах: лямбды вроде `(lab1, lab2) -> lab1.getDifficulty().compareTo(lab2.getDifficulty())` — это реализации стратегии сортировки.
- **Flyweight, Interpreter, Adapter, Facade, Proxy:** напрямую в стандартной архитектуре такой небольшой лабораторной (без использования Spring или сложных графических библиотек) в явном виде не реализуются студентом, но существуют "под капотом" в JVM или более сложных библиотеках (например, Adapter для преобразования интерфейсов IO-потоков).
