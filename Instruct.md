# Инструкция по переходу на Клиент-Серверную архитектуру (Lab 6)

Данный документ содержит подробный план действий для каждого пакета проекта по разделению на клиентский и серверный модули.

## 1. Общие изменения (Project-wide & General)

### `pom.xml`
- [ ] Добавить зависимости для **Log4j2** (core, api, slf4j).
- [ ] Проверить наличие зависимостей для работы с XML (Jackson/JAXB).

### Пакет `com.akira.general.datas`
- [ ] Реализовать интерфейс `java.io.Serializable` во всех классах данных: `LabWork`, `Coordinates`, `Person`, `Location`, `Difficulty`.
- [ ] Убедиться, что `LabWork` имеет корректные методы доступа (getters/setters) для сериализации.

### Пакет `com.akira.general.network`
- [ ] **Класс `Request`**:
    - Должен реализовывать `Serializable`.
    - Поля: `String commandName`, `ArrayList<String> args`, `LabWork objectArgument`.
    - Конструкторы для разных типов команд.
- [ ] **Класс `Response`**:
    - Должен реализовывать `Serializable`.
    - Поля: `String message`, `boolean success`, `Hashtable<Integer, LabWork> collection` (для команд типа `show`).

### Пакет `com.akira.general.commands`
- [ ] Исправить `package com.akira.commands;` на `package com.akira.general.commands;` во всех файлах.
- [ ] **Интерфейс `Command`**:
    - Изменить сигнатуру `execute()`: она должна принимать необходимые зависимости (например, `CollectionManager`) и возвращать `String` или `Response`.
    - Команды НЕ должны использовать `System.out` или `Scanner`.
- [ ] **Интерфейс `Modable`**:
    - Пересмотреть необходимость этого интерфейса. В новой архитектуре аргументы (включая сложные объекты) передаются через `Request`. Возможно, стоит объединить логику в `Command`.
- [ ] Разделить логику команд: на клиенте остается только формирование `Request`, на сервере — выполнение логики над коллекцией.

## 2. Клиентский модуль (`com.akira.client`)

### `Logging и Ресурсы`
- [ ] Создать директорию `src/main/resources`.
- [ ] Добавить конфигурационный файл `log4j2.xml` для настройки логирования на сервере.

### `NetworkManager`
- [ ] Реализовать подключение к серверу по TCP через `Socket`.
- [ ] Методы для отправки `Request` и чтения `Response`.
- [ ] Реализовать механизм обработки временной недоступности сервера (повторные попытки подключения).

### `Validation / Input`
- [ ] Перенести логику чтения объектов из `LabWorkReader` (сервер) в новый класс на клиенте (например, `LabWorkConsoleInput`).
- [ ] Реализовать валидацию всех вводимых данных ПЕРЕД отправкой на сервер.

### `Main` (Client)
- [ ] Интерактивный цикл:
    1. Считать строку.
    2. Распознать команду.
    3. Если команда требует ввода `LabWork` (insert, update), запросить поля у пользователя.
    4. Сформировать объект `Request`.
    5. Отправить на сервер через `NetworkManager`.
    6. Получить `Response` и вывести его содержимое пользователю.
- [ ] Исключить команду `save` из списка доступных клиенту.
- [ ] Команда `exit` должна корректно закрывать соединение и завершать работу клиента.

## 3. Серверный модуль (`com.akira.server`)

### Сетевое взаимодействие (NIO)
- [ ] Создать класс для запуска сервера на базе `ServerSocketChannel`.
- [ ] Настроить неблокирующий режим (`configureBlocking(false)`).
- [ ] Реализовать модули:
    - Приём подключений.
    - Чтение запроса (`Request`).
    - Отправка ответа (`Response`).

### `CommandInvoker` (Server)
- [ ] Хранит реестр доступных команд.
- [ ] Получает `Request`, находит нужную команду и вызывает её `execute()`.
- [ ] Обрабатывает специальную команду `save`, доступную только серверу (через консольный ввод на стороне сервера).

### `CollectionManager`
- [ ] Переписать все методы обработки коллекции с использованием **Stream API** и лямбда-выражений.
- [ ] Метод получения коллекции для клиента должен возвращать элементы, отсортированные по названию.
- [ ] Обеспечить автоматическую генерацию полей (id, creationDate) при добавлении новых объектов.

### `Logging`
- [ ] Интегрировать **Log4j2**.
- [ ] Логировать: запуск сервера, новые подключения, полученные запросы, результаты выполнения команд, ошибки.
